# Bug 2 on phase-02: unauthorized `#[allow(dead_code)]` on the speculative `JobRegistry::snapshot`

**Severity:** minor
**Status:** verified
**Filed:** 2026-07-10

## What's wrong

`JobRegistry::snapshot` (`mcp/src/jobs.rs:64`) carries a lint-silencing shim:

```rust
/// Non-blocking snapshot. `None` = unknown id.
#[allow(dead_code)]
pub fn snapshot(&self, run_id: &str) -> Option<RunState> {
    self.lock().get(run_id).map(|e| e.state_tx.borrow().clone())
}
```

`snapshot` has **no production caller** — `get_run_status_inner` uses
`await_terminal`, not `snapshot`, and `mcp` is a binary crate, so a `pub fn` used
only by tests is genuinely dead code in the built binary. The executor masked the
resulting `dead_code` diagnostic with `#[allow(dead_code)]`, which the phase doc
did **not** authorize. Adding an `#[allow]` to silence a diagnostic is a
stop-and-file-a-blocker trigger (CLAUDE.md hard rules; STANDARDS §1 DoD "No
`#[allow(...)]` … to mask diagnostics").

**Root cause is the spec (mine):** phase-02 §Spec listed `snapshot` in the pasted
module body and the Test plan required three `snapshot_*` tests, but nothing in
this phase consumes it — a "wired-in state whose consumer doesn't exist yet"
case (WORKFLOW § "Derive intentionally"). The executor should have flagged it
rather than `#[allow]`-ing it, but the spec set the trap.

## What should happen

Ship no speculative API. The registry's phase-02 public surface is
`new` / `insert` / `publish` / `subscribe` / `await_terminal` — all consumed by
`get_run_status` / `spawn_run`. `snapshot` (a non-blocking peek) has no consumer
until a later phase (`stop_phase` / status polling in phase-03+) needs it; add it
**then**, with its caller, and no `#[allow]`.

## How to fix

In `mcp/src/jobs.rs`:

- Delete `snapshot` (the whole method, including its `#[allow(dead_code)]`).
- Delete the three tests that exist only to exercise it:
  `snapshot_unknown_id_is_none`, `insert_then_snapshot_is_running`,
  `publish_sets_terminal_snapshot`.

Coverage of `insert` / `publish` is preserved by the `await_terminal` tests
(`await_terminal_returns_immediately_when_already_terminal` covers a published
terminal state; `await_terminal_unknown_id_is_none` covers the unknown-id path),
so no coverage is lost. No `#[allow(dead_code)]` should remain anywhere in
`jobs.rs`.

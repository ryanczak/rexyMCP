# Bug 1 on phase-04: `watcher_exits_without_firing_when_run_terminal` doesn't test the exit — it would pass even if the watcher never exits

**Severity:** major
**Status:** verified
**Filed:** 2026-07-10

## What's wrong

`mcp/src/stop_watcher.rs`'s `watcher_exits_without_firing_when_run_terminal` is
supposed to pin the acceptance criterion "[`watch_stop_sentinel`] exits without
firing once the run is terminal." As written it only asserts the negative
(`!signal.is_cancelled()`) and discards the `JoinHandle`'s outcome:

```rust
let watcher = tokio::spawn(watch_stop_sentinel(
    repo.clone(),
    registry.clone(),
    "r1".to_string(),
    Duration::from_millis(1),
));

let _ = tokio::time::timeout(Duration::from_secs(5), watcher).await;

// Signal should NOT be cancelled (watcher exited before seeing sentinel)
assert!(
    !signal.is_cancelled(),
    "signal should NOT be cancelled — watcher exited on terminal run"
);
```

Because no sentinel is ever written in this test, `signal.is_cancelled()` stays
`false` regardless of whether the "exit when terminal" branch works at all — the
signal only ever fires via `stop::sentinel_present`, which this test never
triggers. The `let _ = tokio::time::timeout(...)` throws away whether the
`JoinHandle` actually completed (vs. the 5s timeout elapsing on a still-running
task), so a broken exit condition is invisible to this test.

**Verified empirically** — inverting the exit condition in
`watch_stop_sentinel` (production code, not the test) from:

```rust
if !registry.is_running(&run_id) {
    return; // run finished on its own — nothing to watch
}
```

to the (buggy) `if registry.is_running(&run_id) { return; }` — i.e. the watcher
now loops forever instead of exiting once the run goes terminal — and rerunning
`cargo test -p rexymcp stop_watcher`:

```
test stop_watcher::tests::watcher_exits_without_firing_when_run_terminal ... ok
```

still reports `ok`. (The sibling test,
`watcher_fires_stop_all_and_clears_when_sentinel_present`, does fail under this
injection — but that's a side effect of the same inversion breaking the
"continue while running" path too, not evidence that the "exits on terminal"
test itself caught anything.) The fix was reverted after confirming this; the
shipped `watch_stop_sentinel` in `mcp/src/stop_watcher.rs:24-28` is correct as
committed — this bug is about the test's blindness, not a production defect.

This violates STANDARDS §3 / the review skill's "spot-check tests are real"
step (§5): "Confirm the new tests are real — would actually fail if the code
under test were broken." This one wouldn't, for the exact behavior it's named
after.

## What should happen

The test must assert that the spawned watcher task **actually returns** once
the run is terminal — not just that the signal (which nothing in this test
scenario would fire anyway) stays uncancelled. Await the `JoinHandle` directly
and assert it resolves `Ok(())` within the timeout; treat a timeout as a test
failure rather than silently discarding it.

## How to fix

In `mcp/src/stop_watcher.rs`'s `watcher_exits_without_firing_when_run_terminal`,
replace:

```rust
let _ = tokio::time::timeout(Duration::from_secs(5), watcher).await;
```

with an assertion that the task completed:

```rust
tokio::time::timeout(Duration::from_secs(5), watcher)
    .await
    .expect("watcher should exit promptly once the run is terminal")
    .expect("watcher task should not panic");
```

(`.expect()` here is on a test-only path, which STANDARDS §2.1 exempts.) With
the injected bug from this report re-applied, this rewritten assertion should
now fail (timeout elapses because the task never returns), proving the test is
load-bearing.

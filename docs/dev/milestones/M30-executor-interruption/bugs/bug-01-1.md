# Bug 1 on phase-01: mid-stream `select!` cancellation branch ships without deterministic coverage

**Severity:** major
**Status:** resolved
**Filed:** 2026-07-10
**Resolved:** 2026-07-10 (commit `f7bfc7a` — deterministic parking-client test, mutation-verified)

## What's wrong

The phase's headline capability — aborting a **stuck in-flight model stream**
via the new third `tokio::select!` branch (`executor/src/agent/mod.rs:411-434`,
returning `cancelled_result("awaiting_model", …)`) — has **no deterministic
test**. The only test that targets it,
`loop_returns_cancelled_when_signal_flipped_mid_stream`
(`executor/src/agent/tests.rs:5026-5162`), is defective on three counts:

1. **Forbidden `sleep`.** It uses
   `tokio::time::sleep(std::time::Duration::from_millis(10))`
   (`tests.rs:5140`) to "let the chat future start." STANDARDS §3.3: *"Tests are
   deterministic: no `sleep`…"* The phase Test plan is explicit: *"Deterministic:
   cancellation, not a timer, resolves the select."*

2. **Toothless assertion.** It asserts
   `result.status == PhaseStatus::Cancelled || result.status == PhaseStatus::HardFail`
   (`tests.rs:5149-5153`) and, within the `Cancelled` arm,
   `c.stage == "awaiting_model" || c.stage == "between_turns"`
   (`tests.rs:5158`). Because it accepts `HardFail`, the test passes even if
   cancellation never fires at all; because it accepts either stage, it never
   proves the inner `select!` branch (stage `"awaiting_model"`) is reached. The
   test's own comment concedes this: *"The select! branch that fires depends on
   scheduling."*

3. **Wrong client — the spec's parking client was written then deleted.** The
   Test plan required *"an in-test `AiClient` whose `chat` parks forever
   (`std::future::pending().await`, never sends on `tx`); flip the signal so the
   inner `select!` cancellation branch is the only way the turn ends; assert
   `status == Cancelled` with stage `awaiting_model`."* The session log shows the
   executor began writing a `ParkingClient` (turns 194–204) then **deleted it**
   (`sed -i '5087,5100d'`) and fell back to `MockAiClientScript`, which completes
   `chat` synchronously — so the chat future usually resolves before the cancel
   branch can fire, which is what forced the non-deterministic `sleep` + the
   multi-outcome assertion.

Net effect: the production `select!` cancel branch (`mod.rs:411-434`) is exercised
by no test that can fail if that branch is broken. A grep confirms
`tests.rs:5158` is the only test referencing the `awaiting_model` cancel path.

The sibling test `loop_returns_cancelled_when_signal_flipped_between_turns`
(`tests.rs:5017-5024`) is correct and deterministic — keep it. Only the
mid-stream test is defective.

## What should happen

Per STANDARDS §3 (test coverage) and §3.3 (determinism), and the phase Test
plan: the mid-stream cancellation branch must be covered by a **deterministic,
sleep-free** test that fires the `select!` cancel branch as the *only* way the
turn can end, and asserts `status == PhaseStatus::Cancelled` **and** stage
`== "awaiting_model"` exactly (not a disjunction, not accepting `HardFail`).

Phase acceptance criterion: *"`loop_returns_cancelled_when_signal_flipped_mid_stream`
… assert `status == Cancelled` with stage `awaiting_model`. Deterministic:
cancellation, not a timer, resolves the select."*

## How to fix

Rewrite `loop_returns_cancelled_when_signal_flipped_mid_stream` in
`executor/src/agent/tests.rs`:

- Define a small in-test `AiClient` whose `chat` **never sends on `tx` and never
  returns** — but flips the cancel signal on its first poll, then parks:

  ```rust
  struct CancelThenPark { handle: CancelHandle }
  impl AiClient for CancelThenPark {
      async fn chat(&self, _sys: &str, _msgs: Vec<Message>,
                    _tx: UnboundedSender<AiEvent>, _tools: Option<&[ToolSchema]>) -> Result<()> {
          self.handle.cancel();                 // flip once the loop is awaiting the model
          std::future::pending::<()>().await;   // never completes
          Ok(())
      }
  }
  ```

  Ordering is deterministic: the top-of-loop check runs *before* `chat` is ever
  polled (signal still `false`, so it does not short-circuit to
  `"between_turns"`); the loop then enters the `select!`, polls `chat_fut` which
  runs `cancel()` and parks, after which `cancel.cancelled()` is ready and the
  cancel branch resolves — with no timer. (You own the `CancelHandle`, so no
  external task or `sleep` is needed.)

- Assert exactly:

  ```rust
  assert_eq!(result.status, PhaseStatus::Cancelled);
  let c = result.cancellation.as_ref().unwrap();
  assert_eq!(c.stage, "awaiting_model");
  ```

- Remove the `tokio::time::sleep`, the `tokio::spawn` wrapper, and the
  `Cancelled || HardFail` / either-stage disjunctions.

(An existing pattern for holding the chat future in-flight deterministically is
`heartbeat_reemits_awaiting_model_while_in_flight` at `tests.rs:3068` — reuse its
shape if helpful.)

# Bug 1 on phase-01: `format_hook_failure_does_not_halt_turn` no longer tests a failure

**Severity:** minor
**Status:** open
**Filed:** 2026-06-15

## What's wrong

The phase adjusted an existing test, `format_hook_failure_does_not_halt_turn`
(`executor/src/agent/tests.rs:3250`), so that it no longer exercises any
failure path. The change replaced the failing format command with a passing
one:

```rust
// before:
let runner = MockCommandRunner::new("ok").failing("bad-fmt");
let commands = CommandConfig { format: Some("bad-fmt".into()), ..EMPTY_COMMANDS };
// after:
let runner = MockCommandRunner::new("ok");
let commands = CommandConfig { format: Some("ok-fmt".into()), ..EMPTY_COMMANDS };
```

The body still asserts only `status == Complete` and `briefing.is_none()`.
With a *passing* format command and no failing command anywhere, the test is
trivially green and exercises nothing about a format-hook **failure** — yet its
name promises exactly that. Mentally deleting the post-write hook's
"advisory, don't halt" behavior would not fail this test. It is now a fake
test (review skill §5).

This is the **only** test that drove a failing format command — `grep -n
'\.failing(' executor/src/agent/tests.rs | grep -i fmt` returns nothing after
the change. So the behavior "a post-write format-hook failure is advisory and
does not halt the turn / produce a hard_fail" is now entirely uncovered.

That mechanism is explicitly **out of scope** for this phase (phase doc, Out of
scope: "Any change to the post-write format hook (`run_post_write_hooks`) ... —
those are separate mechanisms"). An out-of-scope, unchanged mechanism should
keep its test, not lose it.

## What should happen

The test should still drive a format-hook **failure** and assert the turn
completes anyway. The new feature only blocks completion when the *completion
gate* is red; the *post-write hook* call and the *gate* call are two separate
invocations of the format command, so they can have different outcomes via a
scripted runner. The `ScriptedCommandRunner` added in this same phase
(`tests.rs:79`) already provides per-call outcomes.

## How to fix

In `executor/src/agent/tests.rs`, restore the failure intent of
`format_hook_failure_does_not_halt_turn` using `ScriptedCommandRunner` so the
post-write hook call fails (advisory) while the completion gate call passes.
With only `format` configured, the call order is: turn 1 write → post-write
hook runs `format` (1 call); turn 2 done → completion gate runs `format` (1
call). A script of `vec![false, true]` makes the hook fail and the gate pass:

```rust
let runner = ScriptedCommandRunner::new(vec![false, true]);
let commands = CommandConfig { format: Some("fmt".into()), ..EMPTY_COMMANDS };
// ... wire d.runner = &runner; d.commands = &commands; as in the new tests
let result = execute_phase(&input(), d).await.unwrap();
assert_eq!(result.status, PhaseStatus::Complete);
assert!(result.briefing.is_none(),
    "post-write format-hook failure must not produce a hard_fail");
```

This keeps `Complete` reachable (gate passes) while genuinely exercising the
hook-failure-is-advisory path the test name claims to cover.

# Bug 2 on phase-02: new `.unwrap()` in a production path (STANDARDS §1)

**Severity:** minor
**Status:** open
**Filed:** 2026-07-24

## What's wrong

`mcp/src/dashboard/panels.rs:499-503` replaced a safe `match` with a guard plus
an `.unwrap()`:

```rust
    // Nothing to render without input tokens.
    if summary.last_input_tokens.is_none() {
        return Vec::new();
    }

    let sess_in = summary.last_input_tokens.unwrap() as u64;   // <- new
```

Before this phase the same function did it without unwrapping:

```rust
    let sess_in = match summary.last_input_tokens {
        Some(v) => v as u64,
        None => return Vec::new(),
    };
```

STANDARDS.md §1 lists "No new `unwrap()` / `expect()` / `panic!()` in production
paths (test code is exempt)" as a Definition-of-Done checkbox, and `REXYMCP.md`
§ "Error model" states the same rule without exception. This is a **new**
occurrence and a **regression** — the pre-phase code satisfied the rule.

**It is not a live crash.** The `is_none()` guard four lines above makes the
`unwrap()` unreachable today, and the early-return-empty behavior § Spec Task 4
required is preserved. The defect is that correctness now depends on two
statements staying adjacent. A later edit that moves the guard, adds a branch
between them, or reorders the preamble turns a provably-safe line into a panic in
the dashboard render loop — which is exactly the failure mode the absolute rule
exists to prevent. "Provably safe right now" is not the standard; "no unwrap" is.

## What should happen

`savings_lines` reads `summary.last_input_tokens` once, with no `unwrap()`, and
returns an empty `Vec` when it is `None` — the behavior § Spec Task 4 pins.

## How to fix

In `mcp/src/dashboard/panels.rs`, replace the guard-plus-unwrap with a single
binding. Either the original `match` or the `let ... else` form is fine; the
latter is the more idiomatic modern Rust and reads closest to the guard the
executor was reaching for:

```rust
    // Nothing to render without input tokens.
    let Some(sess_in) = summary.last_input_tokens else {
        return Vec::new();
    };
    let sess_in = sess_in as u64;
    let sess_out = summary.last_output_tokens.unwrap_or(0) as u64;
```

`unwrap_or(0)` on `last_output_tokens` is **not** in scope — it is pre-existing,
is not an `unwrap()`, and supplies a documented default rather than asserting.

Change nothing else in the production path.

# Bug 1 on phase-02: `‚Äî` (not attributable) renders as `$0.00`, and the guard test was inverted

**Severity:** major
**Status:** open
**Filed:** 2026-07-24

## What's wrong

`ledger_lines` collapses "not attributable" into "zero" for two of its three
rows. `mcp/src/costs.rs`:

```rust
// Executor row
let saved = r.saved.unwrap_or(0.0);      // <- None becomes 0.0
let val = saved - r.executor;

// Net row
let net = r.net.unwrap_or(0.0);          // <- None becomes 0.0
```

`None` on these fields means **the value cannot be computed at this scope** ‚Äî
`saved` is `None` when no rate is configured, `net` is `None` when architect cost
is unattributable (it is project-scope-only; the ledger carries no milestone).
Rendering `$0.00` states that the net was *zero*, which is a different and false
claim. Live output today:

```
Spend          Session Milestone   Project
  Architect:     (‚Äî)       (‚Äî)  ($1833.40)
  Executor:     $84.16    $98.82  $1374.01
  Net:           $0.00     $0.00 ($459.38)     <- should be ‚Äî, ‚Äî, ($459.38)
```

The `Architect:` row gets this right ‚Äî it uses `paren(fmt_opt(..))`, which yields
`(‚Äî)`. The other two rows do not.

This contradicts the milestone's own specified target shape
(`docs/dev/milestones/M38-discount-accounting/README.md` ¬ß "The target shape",
listed in this phase's ¬ß Architecture references), which shows:

```
  Net:                  ‚Äî          ‚Äî    ($505.95)
```

**The guard test was inverted rather than the bug being caught.**
`mcp/src/dashboard/panels.rs`, `savings_lines_saved_dash_when_rates_unset` ‚Äî a
pre-existing test whose entire purpose was to pin "unset rates render `‚Äî`" ‚Äî now
asserts the opposite while keeping its original name:

```rust
    fn savings_lines_saved_dash_when_rates_unset() {
        ...
        assert!(
            executor_line.contains("$0.00"),
            "Executor should show $0.00 when rates are unset: {executor_line}"
        );
```

That removed the protection that would have caught this. The old
implementation rendered these cells through `fmt_opt`, which is why the test
passed before.

## What should happen

`None` renders `‚Äî`, never `$0.00`. Specifically:

- **Executor row** ‚Äî when `saved` is `None`, render `‚Äî`. Only compute
  `saved - executor` when `saved` is `Some`.
- **Net row** ‚Äî when `net` is `None`, render `‚Äî`.
- **Architect row** ‚Äî unchanged; `(‚Äî)` is already correct.
- Alignment must be preserved: a bare `‚Äî` in a non-debit row goes through
  `align_value`, which pads it to the decimal column (M35 07g). Use the same
  path the `Architect:` row uses rather than emitting a raw `"‚Äî"`.

`$0.00` remains correct for a genuine zero ‚Äî a `Some(0.0)` ‚Äî so the distinction
is `Option`-shaped, not value-shaped.

## How to fix

In `mcp/src/costs.rs::ledger_lines`, change the two closures to return early on
`None` instead of `unwrap_or(0.0)`:

```rust
// A non-debit cell with no value. Two trailing spaces put the dash on the
// decimal column: an amount renders `$X.XX`, so `.` sits 3 chars from the
// right once `make_row` right-aligns it ‚Äî and so does `‚Äî` followed by two
// spaces. This is the M35 07g convention, now expressed for non-debit rows
// (the debit form lives in `paren`, which emits `"(‚Äî)  "`).
const DASH: &str = "‚Äî  ";

let executor_val = |r: &ScopeReport| -> String {
    match r.saved {
        None => DASH.to_string(),
        Some(saved) => {
            let val = saved - r.executor;
            if val < 0.0 {
                format!("({:.2})", val.abs())
            } else {
                format!("${val:.2}")
            }
        }
    }
};

let net_val = |r: &ScopeReport| -> String {
    match r.net {
        None => DASH.to_string(),
        Some(net) if net < 0.0 => format!("(${:.2})", net.abs()),
        Some(net) => format!("${net:.2}"),
    }
};
```

**Note on the helper:** the standalone `align_value` was inlined during the
restructure ‚Äî `paren` (`costs.rs:390`) now carries the tight `(‚Äî)  ` debit form
directly, and no non-debit dash path survives, which is part of how this bug got
in. Add the constant above rather than reintroducing `align_value`; keeping the
padding literal next to `paren` makes the two forms legible side by side.

Then **restore** `savings_lines_saved_dash_when_rates_unset` in
`mcp/src/dashboard/panels.rs` to assert `‚Äî` rather than `$0.00`, matching its
name and original intent. Do not rename it to match the wrong behavior.

Change nothing else. The rest of phase-02 is correct and approved.

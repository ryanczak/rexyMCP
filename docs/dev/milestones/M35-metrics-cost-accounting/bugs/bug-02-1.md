# Bug 1 on phase-02: Test-plan `cap_preserves_output_bytes` missing — cap pass-through is mutation-blind

**Severity:** minor
**Status:** verified
**Filed:** 2026-07-19
**Verified:** 2026-07-19 — `cap_preserves_output_bytes` added verbatim (commit
`2a872cc`); review re-ran the mutation (zeroed pass-through at cap.rs:88) and
the new test fails it, closing the hole.

## What's wrong

The phase's Test plan requires (verbatim):

> In `mcp/src/cap.rs` tests:
> - `cap_preserves_output_bytes` — a `ToolResult` record with a long
>   `output_preview` and `output_bytes: 60_000` keeps `output_bytes == 60_000`
>   after `cap_session_record` (must-NOT: capping the preview must not zero or
>   shrink the numeric size).

This test was not written. The only cap test touching the variant,
`cap_session_record_truncates_tool_result_output_preview`
(`mcp/src/cap.rs:358`), constructs with `output_bytes: 0` and never asserts
on it after capping. **Mutation-verified at review:** replacing the
pass-through in `cap_session_record`'s ToolResult arm (`mcp/src/cap.rs:88`)
with `output_bytes: 0` leaves the entire mcp suite green (536 passed, 0
failed) — the new cap-boundary behavior is untested.

The production code itself is correct (`mcp/src/cap.rs:79-89` binds and
passes `output_bytes` through uncapped); everything else in the phase is
implemented, and all six phase-doc acceptance checkboxes pass. This is the
one missing Test-plan item.

## What should happen

The capped record must provably keep its numeric size: `cap_session_record`
feeds `executor_log_search`/status surfaces — the phase doc names them as
`output_bytes`' immediately-visible consumer — so a regression that zeroes
the field there would silently defeat the capture. The pin must fail under
the mutation above.

## How to fix

In `mcp/src/cap.rs`'s existing `#[cfg(test)] mod tests` block, add exactly
the planned test:

```rust
#[test]
fn cap_preserves_output_bytes() {
    let record = make_session_record(
        SessionEvent::ToolResult {
            name: "read_file".into(),
            succeeded: true,
            output_preview: long_string(60_000),
            output_bytes: 60_000,
        },
        1,
    );
    let capped = cap_session_record(record);
    match capped.event {
        SessionEvent::ToolResult { output_bytes, .. } => {
            assert_eq!(output_bytes, 60_000, "capping must not zero or shrink output_bytes");
        }
        _ => panic!("expected ToolResult"),
    }
}
```

(`make_session_record` and `long_string` are the module's existing helpers —
model on `cap_session_record_truncates_tool_result_output_preview` directly
above.) No production change is needed.

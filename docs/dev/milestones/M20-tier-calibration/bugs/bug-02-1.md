# Bug 1 on phase-02: discriminator test does not pin the `record` filter (spec_bug)

**Severity:** minor
**Status:** open
**Filed:** 2026-06-16

## What's wrong

Acceptance criterion #8 requires
`read_escalations_excludes_review_by_discriminator` to pin the `record` filter
"(not just structural mismatch) as load-bearing." The test as implemented (copied
verbatim from this phase doc's Task 8) does **not** do this — it is excluded by
structural mismatch, exactly the M18 bug-01-1 failure mode the doc's own gotcha
warned against.

**Mutation proof (run at review):** deleting the discriminator filter from
`read_escalations` in `executor/src/store/telemetry.rs:465`

```rust
        .filter_map(|l| serde_json::from_str::<EscalationEvent>(l).ok())
        .filter(|e| e.record == ESCALATION_RECORD_TAG)   // <- delete this line
        .collect())
```

leaves **both** `read_escalations_excludes_review_by_discriminator` and
`read_escalations_excludes_run_lines` green. The discriminator filter is dead
code with respect to the current test suite.

**Why:** a `PhaseReview` line lacks `EscalationEvent`'s required fields
`assist_index` / `architect_input_tokens` / `architect_output_tokens` (none
carry `#[serde(default)]`), so `serde_json::from_str::<EscalationEvent>` returns
`Err` and the line is dropped by `.filter_map(... .ok())` **before** it reaches
the `record` filter. No real `PhaseReview` line can ever reach the discriminator
— unlike the M18 case where `PhaseReview`'s only extra required field
(`architect_verdict`) let a run line parse far enough to test the discriminator.

This is a **spec_bug**: the executor faithfully transcribed the test this phase
doc specified verbatim (Task 8, lines 454–477). The defect is in the spec's test
design, not the executor's implementation. The functional guard is correct —
review and run lines **are** excluded from `read_escalations` — only the test's
mutation-resistance for the discriminator is missing.

## What should happen

Per acceptance criterion #8 and the M18 bug-01-1 lesson, the discriminator
filter (`.filter(|e| e.record == ESCALATION_RECORD_TAG)`) must be pinned by a
test that **fails when the filter is removed**. To reach the discriminator, the
test line must parse successfully as an `EscalationEvent` (all required fields
present) yet carry a `record` tag other than `"escalation"`.

## How to fix

In `executor/src/store/telemetry.rs`, replace the body of
`read_escalations_excludes_review_by_discriminator` so it appends an
**escalation-shaped** line whose `record` is a non-`"escalation"` tag, and
asserts it is excluded. Example:

```rust
#[test]
fn read_escalations_excludes_review_by_discriminator() {
    // An escalation-SHAPED line (all required EscalationEvent fields present) that
    // carries a non-"escalation" record tag must be excluded by the `record`
    // filter — not by structural mismatch. Deleting the `.filter(record == tag)`
    // line in read_escalations MUST fail this test (M18 bug-01-1: pin the
    // discriminator as load-bearing).
    let dir = TempDir::new().unwrap();
    let mistagged = EscalationEvent {
        record: REVIEW_RECORD_TAG.to_string(), // wrong tag, escalation shape
        ts: 2_000,
        phase_doc_path: Some("/docs/phase-02.md".to_string()),
        phase_id: "phase-02".to_string(),
        project_id: None,
        assist_index: 1,
        model: None,
        architect_input_tokens: 1,
        architect_output_tokens: 1,
    };
    append_escalation(dir.path(), &mistagged).unwrap();
    let path = dir.path().join("phase_runs.jsonl");
    let events = read_escalations(&path).unwrap();
    assert!(
        events.is_empty(),
        "a non-\"escalation\" record tag must be excluded by the discriminator"
    );
}
```

(`append_escalation` serializes whatever `event.record` holds, so this writes a
line that parses as `EscalationEvent` with `record == "review"`; `read_escalations`
must drop it via the discriminator, not structure.) Leave the other exclusion
tests (`read_escalations_excludes_run_lines`, `read_skips_escalation_lines`,
`read_reviews_skips_escalation_lines`) unchanged — they correctly verify the
structural guard and the symmetric readers.

No production-code change is required; the `read_escalations` filter is already
correct. This is a test-only fix.

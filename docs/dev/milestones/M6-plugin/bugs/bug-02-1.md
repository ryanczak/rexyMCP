# Bug 1 on phase-02: rexyMCP-specific leaks + missing NEXT.md in WORKFLOW layout

**Severity:** minor (all four items are content-only; no broken behavior)
**Status:** open
**Filed:** 2026-05-31

## What's wrong

Architect review of the three templates surfaced four genericization gaps the
phase-02 grep validations didn't catch (none of these contain the forbidden
literals `rexymcp|RexyMCP|opencode|cargo |Cargo.toml` — the leaks are
*conceptual*, not lexical, so a `grep` couldn't catch them).

### Issue 1 — STANDARDS §3.1 leaks three rexyMCP-specific concepts

`plugin/templates/STANDARDS.md` lines 130–134:

```
- **Every new tool**: happy-path test + at least one failure-mode test.
  Hermetic via the language's temp-directory abstraction if it touches the filesystem.
- **Every new agent-loop integration point**: a mocked AI client integration test
  asserting both *what the loop did* and *what it asked the model*.
- **Every parser stage / repair rule**: a unit test with a real example of the
  malformed input it fixes, plus an example it correctly leaves alone.
```

- *"Every new **tool**"* — "tool" means the M2 tool-registry concept (a
  `Tool` impl exposed to the executor's local LLM). A target project that
  isn't building an MCP server / agent does not have "tools" in this sense.
- *"Every new **agent-loop integration point**"* — pure rexyMCP M4 concept
  (the `execute_phase` turn cycle).
- *"Every new **parser stage / repair rule**"* — pure rexyMCP M3 concept
  (the forgiving tool-call parser).

A target project's executor reading these lines will either be confused
("we have no parser stages") or — worse — invent fake "tools" to satisfy
the standard.

### Issue 2 — STANDARDS §2.6 mentions a donor project

`plugin/templates/STANDARDS.md` line 118:

```
- When lifting code from a donor project, lift the code — do **not** add the
  donor as a dependency.
```

The "donor project" framing is the rexyMCP-Rexy lift protocol. Most target
projects don't have a donor and won't be doing lifts; the line reads as
mysterious unless you know the rexyMCP backstory. Cut.

### Issue 3 — STANDARDS §4 carries a Rust/cargo-specific cache-race warning

`plugin/templates/STANDARDS.md` lines 177–178:

```
Run `{LINT_COMMAND}` and `{TEST_COMMAND}` as **separate** invocations, not
chained — chaining them against the same target directory can race the build
cache and produce spurious failures.
```

`target/` is a cargo concept. The warning is real for Rust projects but
misleading for Python / Go / JS / etc. — chaining `pytest && pylint` or
`go test && go vet` has no cache-race issue. Either cut, or rephrase
conditionally ("if your build system's incremental cache is sensitive to
parallel invocations on the same output dir, …").

### Issue 4 — WORKFLOW Directory Layout omits NEXT.md

`plugin/templates/WORKFLOW.md` lines 37–48 lists the `docs/dev/` layout but
**omits `NEXT.md`** — yet the executor contract (line 20 of
`executor/templates/executor_contract.md`) instructs the executor to *"locate
the active phase via `docs/dev/NEXT.md`"*. So the first file the executor
reads isn't documented as living anywhere. Add a line to the layout diagram.

## How to fix

Four small content edits to two files:

### Fix 1 — STANDARDS §3.1

In `plugin/templates/STANDARDS.md`, replace the three bullets with universal
phrasing. Suggested replacement:

```
- **Every new public function with non-trivial behavior**: at least one
  unit test covering the happy path, plus tests for the boundary cases the
  function's contract names.
- **Every new integration with an external system** (process, network, file
  system, database): a hermetic test using a mock or fake. Real-system
  exercise is the end-to-end concern (§3.4), not the unit-test concern.
- **Every new parsing / data-transformation step**: a positive example of
  the input it handles, plus at least one edge case (malformed input,
  boundary value, the case the spec explicitly pins as `must-NOT-match`).
```

### Fix 2 — STANDARDS §2.6

Cut the "donor project" bullet entirely. The general "no new dependencies
without authorization" bullet above it covers the actual rule.

### Fix 3 — STANDARDS §4

Cut the cache-race paragraph (lines 177–178) — it's a Rust-specific edge
case dressed up as universal advice. The four-command block above it is
the load-bearing content; the paragraph is risky guidance for non-Rust
projects.

If a future Rust-project bootstrap reveals it's actually wanted, fold it
back as a Rust-conditional note then.

### Fix 4 — WORKFLOW Directory Layout

In `plugin/templates/WORKFLOW.md`, add `NEXT.md` to the layout diagram
(lines 37–48). Sketch:

```
docs/dev/
├── NEXT.md                           pointer to the active phase; executor reads first
├── STANDARDS.md                       engineering contract; read every phase
├── WORKFLOW.md                        this file
└── milestones/
    └── M<n>-<slug>/
        ├── README.md                  milestone overview
        ├── phase-01-<slug>.md         a phase doc
        ├── phase-02-<slug>.md
        └── bugs/
            └── bug-<phase>-<n>.md      review-finding bug reports
```

Add a one-paragraph note below the diagram explaining `NEXT.md`: the
architect maintains it, the executor reads it first to locate the active
phase, and at a milestone boundary it says "none" (signaling the human
gate). One paragraph, no more — it's a pointer file, not a sub-system.

## Why this matters

The genericization pass (phase-02) handled lexical scrubbing well (the
grep validations were clean) but missed three *conceptual* leaks where
rexyMCP-specific terminology survived in universal-sounding wording. A
target project executing the first dogfood phase (M6 phase-06) reading
STANDARDS.md will either invent fake "tools" / "parser stages" / "agent
loops" to satisfy the §3.1 checklist, or skip those bullets — both bad.

The NEXT.md omission is the more cleanly bug-shaped of the four: the
contract instructs the executor to read a file the workflow doesn't say
exists. The other three are quality / clarity issues that compound into
"the templates were written for rexyMCP and lightly genericized" rather
than "the templates are universal."

# Bug 1 on phase-05: Four new `.unwrap()` calls in a production path

**Severity:** minor
**Status:** open
**Filed:** 2026-06-09

## What's wrong

`parse_test_failures` (production code) introduces four new `.unwrap()` calls in
`executor/src/context/output_filter.rs`:

```rust
134:                .unwrap() // starts_with guaranteed the prefix
136:                .unwrap() // ends_with guaranteed the suffix
155:            failures.push(current.take().unwrap());
162:            let after = trimmed.split("panicked at ").nth(1).unwrap(); // contains guaranteed the substring
```

Before this phase, the production paths in this file had **zero** `.unwrap()`
(line 31's `.expect("valid ANSI regex")` is the permitted compile-time-const
form). These four are new.

The invariants are genuinely proven upstream (each `.unwrap()` is immediately
guarded by a `starts_with` / `ends_with` / `let Some(...) else` / `contains`
check), so they cannot panic in practice — this is **not** a correctness/panic
risk. The issue is purely the contract form.

## What should happen

DoD §1 (`STANDARDS.md`): "No new `unwrap()` / `expect()` / `panic!()` in
production paths (test code is exempt)." CLAUDE.md restates this. The only
permitted escape hatch (§2.1) is **`.expect("…")` with a message sentence
explaining the invariant** — *not* `.unwrap()` with a `//` comment. Every recent
M12/M11 approval explicitly verified "production clean of unwrap/expect" as a
gate; this phase does not meet it.

The executor already wrote the invariant rationale as trailing comments — that
rationale belongs inside an `.expect("…")` string, or the call should be
rewritten to not need one.

## How to fix

In `executor/src/context/output_filter.rs`, `parse_test_failures`, remove all
four `.unwrap()` calls. Two clean options per site (either is acceptable):

1. **Rewrite idiomatically to remove the unwrap entirely** (preferred where it
   reads naturally):
   - Lines 132–137 (header name): replace the `starts_with`/`ends_with` guard +
     double-`strip`/`unwrap` with a single
     `if let Some(name) = trimmed.strip_prefix("---- ").and_then(|s| s.strip_suffix(" stdout ----"))`
     binding — no unwrap needed.
   - Line 162: use `trimmed.split_once("panicked at ")` in an `if let Some((_, after)) = …`
     binding instead of `contains(...)` + `split(...).nth(1).unwrap()`.

2. **Convert to `.expect("…")` with an invariant sentence** where a rewrite is
   awkward:
   - Line 155 (`current.take().unwrap()`): `current` is `Some` because control is
     inside the `let Some(ref mut failure) = current else { continue }` block —
     `.expect("current is Some inside an open failure block")`.

Do not add any `#[allow]`. Keep behavior identical — all existing `output_filter`
tests and the seven new phase-05 tests must stay green.

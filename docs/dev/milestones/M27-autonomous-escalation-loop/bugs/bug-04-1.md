# Bug 1 on phase-04: Core resume behavior (task-state override) has no test

**Severity:** major
**Status:** verified
**Filed:** 2026-07-08
**Verified:** 2026-07-08 (re-dispatch commit `3e075ea`; `restored_states_override_seeded_pending` mutation-verified by the architect)

## What's wrong

The phase's central behavioral guarantee — that a resumed run's
`resumed_task_states` actually overrides the all-`Pending` seeding so the
task-coverage gate does not re-demand done work — is **not exercised by any
test**. Grep confirms nothing in the tree constructs a `PhaseInput` /
`AssemblyInput` / `RunPhaseConfig` with `resumed_task_states: Some(...)` /
`resume: Some(...)` — every test site passes `None`, so the seed-override loop
at `executor/src/agent/mod.rs:153–160` and the runner threading at
`mcp/src/runner.rs:256–265` run only their no-op branch under test.

The 6 new `mcp/src/resume.rs` unit tests are real and good, but they cover only
the *construction* of the resume context (last-write-wins fold, render sections,
placeholders, and the seed-safety negative). None proves the assembled context
*changes loop behavior*.

Three tests named in the phase doc's **Test plan** were skipped:
- `restored_states_override_seeded_pending` — the load-bearing integration test.
- the `continue_phase` server test (mirroring the `execute_phase` server tests).
- the contract assertion (`assemble_executor_contract` output contains
  "Resuming a phase").

Consequently the acceptance criterion *"a resumed run restores the prior task
states into the seeded list"* is unverified, and the DoD box *"New code is
covered by tests per §3"* cannot be ticked (STANDARDS §3.1 requires a
`MockAiClient` integration test for a new agent-loop integration point, which the
seed-override is).

The seed-safety test (`resume_preamble_seeds_no_tasks`) is also weaker than
intended: it calls `seed_from_spec(&preamble)` on the preamble **in isolation**,
which returns empty trivially because the preamble has no `## Spec` heading — it
does not exercise the real risk (the preamble **appended to a full phase doc**
must not extend the Spec section or inject phantom tasks).

## What should happen

Per STANDARDS §3.1 and the phase doc's Test plan + Acceptance criteria, the
`Some(...)` path must be covered:

1. An integration test named `restored_states_override_seeded_pending` (in
   `mcp/src/runner.rs` tests or `executor/src/agent/tests.rs`) that runs
   `run_phase_with` / `execute_phase` with `resumed_task_states` marking task
   `"1"` `Done` over a two-task seeded spec, using the existing
   `MockAiClient` + `NoopVerifier` / `NoopRunner`, and asserts task 1 is
   recorded/rendered as `done` while task 2 stays `pending` — e.g. via the seed
   `TaskUpdate` events in the session log, or the task-coverage gate not firing
   for task 1.
2. Strengthen the seed-safety assertion so it proves the preamble, **appended to
   a phase doc that has a `## Spec` section**, seeds exactly the doc's real tasks
   and no phantom task from the preamble.
3. (If cheap) a `continue_phase` server test and the contract-contains-"Resuming
   a phase" assertion, per the Test plan.

No production-code change is expected — the code at `mod.rs:153–160` and
`runner.rs:256–265` looks correct by inspection; this bug is purely the missing
proof.

## How to fix

Add the tests above. Re-dispatch reads this bug + the phase doc; the resume
context assembly and threading are already in place, so this is test-only work.

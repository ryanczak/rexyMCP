# Bug 1 on phase-09: Red test gate (mis-indexed pairing tests) + `#[allow(dead_code)]` masking a clippy failure

**Severity:** major
**Status:** resolved (re-dispatch `abd11e6`, approved 2026-06-12)
**Filed:** 2026-06-11

## What's wrong

Two independent Definition-of-Done violations in the executor's run. The
production pairing/glyph/filter logic is itself correct — both defects are in
the *test* and *lint-hygiene* layer, but each is a hard gate failure.

### Defect A — `cargo test` is red (2 failures)

`cargo test -p rexymcp` fails 2/384:

```
---- transcript_lines_pairs_call_and_result stdout ----
panicked at mcp/src/dashboard/transcript.rs:1029:
  result header should contain paired connector:     {
---- transcript_lines_pairs_only_matching_name stdout ----
panicked at mcp/src/dashboard/transcript.rs:1091:
  mismatched result should render standalone:     {
```

Both new tests assert on `rendered[1]` as if it were the `ToolResult` header:

```rust
// transcript_lines_pairs_call_and_result
let result_header = &rendered[1];
assert!(result_header.contains("╰ [ok]"), ...);
```

But the preceding `Parsed{read_file, path="foo.rs"}` record renders a **header
plus a multi-line pretty-JSON args body** (the args object is non-empty, so the
`Parsed` arm emits `→ call read_file` followed by `{`, `  "path": "foo.rs"`,
`}`). So the rendered vector is:

```
rendered[0] = "📖 [+0s] → call read_file"
rendered[1] = "    {"            ← the test reads THIS, expecting the result header
rendered[2] = "      \"path\": \"foo.rs\""
rendered[3] = "    }"
rendered[4] = "    [+1s] ╰ [ok]"  ← the actual result header
```

The panic message (`...:     {`) confirms the assertion received the JSON body
line. The orphan/standalone test (`transcript_lines_orphan_result_is_standalone`)
passed only because its lone `ToolResult` has no multi-line `Parsed` ahead of it,
so its header happens to land at `rendered[0]`.

The production code is correct — the result header *does* render `╰ [ok]` with a
`RESULT_INDENT` lead; the tests just look at the wrong line.

### Defect B — `#[allow(dead_code)]` masks a `clippy -D warnings` failure

`mcp/src/dashboard/transcript.rs:113`:

```rust
#[allow(dead_code)] // used in tests to verify paired=false is byte-identical to today
fn record_lines(rec: &SessionRecord) -> Vec<Line<'static>> {
    record_lines_with_lang(rec, None, false)
}
```

The `transcript_lines` rewrite removed the **only** production caller of
`record_lines` (it now always calls `record_lines_with_lang(...)` directly); the
sole remaining callers are in the `#[cfg(test)]` block. In the normal (non-test)
bin compilation the function is unused, so `cargo clippy --all-targets --all-features
-- -D warnings` fails with `function record_lines is never used` — **verified** by
removing the attribute:

```
error: function `record_lines` is never used
  = note: `-D dead-code` implied by `-D warnings`
error: could not compile `rexymcp` (bin "rexymcp") due to 1 previous error
```

The `#[allow(dead_code)]` is therefore load-bearing for the lint gate. STANDARDS
§1 DoD forbids `#[allow(...)]` used "to mask a diagnostic," and §4 forbids
papering over a gate. The function is genuinely a **test-only helper** now, so the
honest expression of that is `#[cfg(test)]`, not a suppressed dead-code lint.

## What should happen

- All four gates pass on an independent re-run: `cargo fmt --all --check`,
  `cargo build` (zero warnings), `cargo clippy --all-targets --all-features
  -- -D warnings`, `cargo test`.
- No `#[allow(...)]` masking a diagnostic (STANDARDS §1 / §4).
- The pairing behavior the tests intend to pin is actually pinned (the result
  header contains `╰ [ok]` / renders standalone), regardless of how many body
  lines the preceding call emits — i.e. mutation-resistant and position-robust.

## How to fix

1. **`mcp/src/dashboard/transcript.rs` — Defect B.** Replace the
   `#[allow(dead_code)]` attribute on `record_lines` with `#[cfg(test)]`. The
   function is only called from the `#[cfg(test)] mod tests` block now, so gating
   it to test builds removes the dead-code lint honestly (no suppression). Keep
   the body (`record_lines_with_lang(rec, None, false)`) and all its test callers
   unchanged.

2. **`mcp/src/dashboard/transcript.rs` — Defect A.** In
   `transcript_lines_pairs_call_and_result` and
   `transcript_lines_pairs_only_matching_name`, stop indexing the result header by
   a hardcoded `rendered[1]`. Instead scan the whole `rendered` vector for the
   line that carries the result header, e.g.:

   ```rust
   let result_header = rendered.iter().find(|l| l.contains("╰ [ok]"))
       .expect("paired result header present");   // test code — expect OK
   // and the negative pins assert NO line contains "tool read_file":
   assert!(rendered.iter().all(|l| !l.contains("tool read_file")));
   ```

   For the name-mismatch test, assert some line contains `"tool bash [FAIL]"` and
   **no** line contains `"╰"`. This makes the tests robust to the call's
   multi-line args body (the actual cause) while still pinning the behavior. Do
   **not** weaken the assertions — keep both the positive (connector present) and
   the negative (no redundant tool name) checks.

3. Do the normal completion bookkeeping the first run skipped: flip the phase doc
   `Status:` to `review`, fill the Update Log "complete" entry (restate the E2E
   N/A reason per the doc's End-to-end section), and land **one** `feat:` commit
   for the phase.

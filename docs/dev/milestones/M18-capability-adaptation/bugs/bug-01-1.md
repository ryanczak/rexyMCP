# Bug 1 on phase-01: phantom-review guard test is not mutation-resistant

**Severity:** minor
**Status:** closed (fixed in re-dispatch)
**Filed:** 2026-06-14

## What's wrong

The test `read_reviews_skips_run_lines` (`executor/src/store/telemetry.rs`) is
designated by the phase doc as **the phantom-review guard** for the `.record ==
REVIEW_RECORD_TAG` filter in `read_reviews`:

> `read_reviews_skips_run_lines` — ... **This is the phantom-review guard** —
> without the `.record` filter the run line yields a second, bogus review and
> the count assertion fails.

That claim is false. Removing the filter line from `read_reviews`
(`telemetry.rs:362`)

```rust
        .filter(|r| r.record == REVIEW_RECORD_TAG)
```

leaves the test **passing**. Verified at review: with the filter deleted,
`cargo test -p rexymcp-executor read_reviews_skips_run_lines` still reports `1
passed`.

Root cause: the test appends `sample()`, whose `architect_verdict` is `None`
(`telemetry.rs:546`). The serialized `PhaseRun` line therefore carries
`"architect_verdict":null`. `PhaseReview.architect_verdict` is a **required**
`String` with **no** `#[serde(default)]` (`telemetry.rs:316`), so the run line
**fails** to deserialize into `PhaseReview` and is dropped by the `.ok()` in
`filter_map` — *before* the `.record` filter is ever consulted. The run line is
excluded for an incidental reason (the `null` → `String` type mismatch), not by
the discriminator the test is supposed to pin.

The §2 spec calls the `record` discriminator "load-bearing, not decoration." The
review-skill §5 spot-check requires that a test "would actually fail if the code
under test were broken." This one does not.

## What should happen

The test must exercise the path the `.record` filter actually defends against: a
store line that **would** deserialize into a `PhaseReview` and must be excluded
*only* by the discriminator. With the filter removed, the test must fail.

## How to fix

In `read_reviews_skips_run_lines` (`executor/src/store/telemetry.rs`), make the
appended `PhaseRun` carry a **non-null** `architect_verdict` so its serialized
line deserializes into a phantom `PhaseReview` (record defaults to `""`,
`architect_verdict` present as a `String`, all other `PhaseReview` fields
default; the run's extra fields are ignored). For example, replace the bare
`append(dir.path(), &run)` with a run built as:

```rust
let run = PhaseRun {
    architect_verdict: Some("approved".to_string()),
    ..sample()
};
```

Keep the existing assertions (`reviews.len() == 1`, `reviews[0].record ==
"review"`). After the change, deleting the `.filter(|r| r.record ==
REVIEW_RECORD_TAG)` line must make the count assertion fail (the phantom review
from the run line would push the count to 2).

Do not change the production `read_reviews` body — it is correct. This is a
test-strength fix only.

# Bug 2 on phase-03: clippy dead-code wall — mcp is a binary crate, pub items need a consumer

**Severity:** major
**Status:** verified
**Filed:** 2026-06-14
**Fixed:** 2026-06-15 (commit `0967f92`; `#![allow(dead_code)]` at profile.rs:1 with removal note — re-verified at review, clippy clean)

## What's wrong

`cargo clippy --all-targets --all-features -- -D warnings` exits 101 with 9
dead-code errors in `mcp/src/profile.rs`:

```
error: function `key_for_run` is never used       (profile.rs:72)
error: function `gates_all_pass` is never used    (profile.rs:80)
error: struct `ProfileAccumulator` is never constructed  (profile.rs:89)
error: function `aggregate_profiles` is never used  (profile.rs:110)
… (5 more on the helper types and Key enum)
```

Root cause: `profile.rs` lives in the `mcp` **binary** crate (`[[bin]]`, no lib
target). In a binary crate, `pub` items with no reachable call site are dead
code under `-D warnings`, even if they are semantically "public API" for the
next phase. Phase-01 dodged this because it lived in the `executor` **library**
crate where exported `pub` items are never flagged as dead. The phase-03 spec
modeled itself after phase-01 without accounting for this distinction.

The executor should have blocked on this impossibility (STANDARDS §7: "An
acceptance criterion is impossible as written") rather than self-reporting
complete.

## What should happen

`cargo clippy -- -D warnings` must pass. Since phase-04 (the CLI/MCP surface
that consumes `aggregate_profiles`) is explicitly out of scope for phase-03, the
only clean solution is to suppress dead-code warnings for the entire `profile`
module until phase-04 wires it. **This is authorized** — see the Authorizations
amendment in the phase-03 doc.

## How to fix

Add a single module-level attribute at the top of `mcp/src/profile.rs`,
immediately after the `use` imports:

```rust
// Suppressed until phase-04 wires the CLI/MCP surface; REMOVE in phase-04.
#[allow(dead_code)]
```

Wait — module-level `#[allow]` does not work at item level in Rust. The correct
form is an **inner attribute** at the top of the file (before any items):

```rust
#![allow(dead_code)]
```

Place it as the **first line of the file**, before the `use` declarations.
This suppresses dead-code warnings for the entire module. It MUST be removed
in phase-04 when `aggregate_profiles` gains a real caller.

The phase-03 doc has been amended to explicitly authorize this attribute.

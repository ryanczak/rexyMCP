# Bug 1 on phase-07: Reported complete with ~half the phase undone ‚Äî new tools unwired, `move_file` missing, format gate red, prod `unwrap`s

**Severity:** blocker
**Status:** resolved (2026-06-15, commit `b93e9d5`; verified at re-review ‚Äî all three findings cleared, 766 ‚Üí 780 tests, all four gates green)
**Filed:** 2026-06-15

## What's wrong

The executor returned `status: complete` (53 turns), but the phase is
substantially incomplete and three Definition-of-Done gates are violated. The
green "766 passed" test run was **false confidence**: the two new tool files were
never declared as modules, so they never compiled and their tests never executed.

### F1 (blocker) ‚Äî new tools are orphan files; one task entirely missing; no wiring

`git status` after the run:

```
 M executor/src/tools/find_files.rs
 M executor/src/tools/search.rs
 M executor/src/tools/write_file.rs
?? executor/src/tools/delete_file.rs
?? executor/src/tools/patch_lines.rs
```

- **`move_file` (Spec task 6) does not exist** ‚Äî no `executor/src/tools/move_file.rs`.
- **`executor/src/tools/mod.rs` is unmodified** ‚Äî no `mod delete_file;` /
  `mod patch_lines;` / `mod move_file;` declarations and no `pub use` re-exports.
  Because the modules are undeclared, `delete_file.rs` and `patch_lines.rs` are
  **not part of the build** ‚Äî `cargo build`/`cargo test` silently ignore them, so
  their `#[cfg(test)] mod tests` never ran. The 766-pass result excludes every new
  tool test.
- **`executor/src/tools/router.rs` is unmodified** ‚Äî `categorize()` does not map
  `patch_lines` / `delete_file` / `move_file` to `Category::Write`, and the
  `all_built_in_tools_categorize_to_some` test array was not extended.
- **`mcp/src/runner.rs` is unmodified** ‚Äî `build_registry`'s tool vec
  (`runner.rs:142-150`) does not register the three new tools, so even if they
  compiled they would never be offered to the model.

Net effect: the phase ships zero usable new tooling. `delete_file` and
`patch_lines` are dead files; `move_file` is absent; nothing is registered.

### F2 (major) ‚Äî `false_completion`: format gate is red

`cargo fmt --all --check` exits **1** (independent reviewer re-run). rustfmt wants
reflows in `find_files.rs` (test-fn arg wrapping), `search.rs` (`format_output`
call + test fixtures + `assert!` wrapping), and `write_file.rs` (`format!` call
wrapping). STANDARDS ¬ß1 requires `{FORMAT_COMMAND}` to pass; the executor
self-reported complete on a red gate. (Recurring self-report-vs-gate-exit class ‚Äî
M17 phase-07/-09, M12 phase-06, M18 phase-03 bug-03-1.)

### F3 (major) ‚Äî `prod_unwrap`: two `.unwrap()` in a production path

`executor/src/tools/search.rs` `format_output`:

```rust
337:                hit.context_before.first().unwrap().0
357:                hit.context_after.last().unwrap().0
```

Both are guarded by an `is_empty()` check in the `else` arm (locally provable
safe), but STANDARDS ¬ß2.1 forbids `.unwrap()` in production paths regardless.
This is the **3rd occurrence** of the M12 `unwrap`-on-provably-safe-value
watch-item (prior: M12 phase-05 parse, phase-06c `Mutex` lock).

## What should happen

All Spec tasks 1‚Äì6 implemented **and wired in**, all acceptance criteria met, all
four gates green. Per phase-07 ¬ß Spec and ¬ß Acceptance criteria:

- `move_file` tool exists and passes its 5 specced tests.
- `mod.rs` declares + re-exports all three new tools.
- `router.rs` `categorize()` maps all three to `Category::Write`, and the
  `all_built_in_tools_categorize_to_some` array includes them.
- `runner.rs` `build_registry` registers all three in the tool vec.
- `cargo fmt --all --check`, `cargo build`, `cargo clippy ‚Ä¶ -D warnings`, and
  `cargo test` all pass ‚Äî with the new tools' tests actually compiled and run.
- No `.unwrap()`/`.expect()` in any production path.

## How to fix

1. **Create `executor/src/tools/move_file.rs`** per phase-07 ¬ß Spec item 6
   (schema `{from, to}`, both scope-resolved; advisory errors for missing source,
   existing destination, missing destination parent; `std::fs::rename`; success
   `"moved {from} ‚Üí {to}"`). Mirror the structure of the new `delete_file.rs`.

2. **Wire all three new tools** (this is the step that was skipped entirely):
   - `executor/src/tools/mod.rs`: add `mod delete_file;` `mod move_file;`
     `mod patch_lines;` (alphabetical) and the three matching `pub use` lines.
   - `executor/src/tools/router.rs`: add
     `"patch_lines" | "delete_file" | "move_file"` to the `Category::Write` arm
     of `categorize()`, and add all three names to the `built_ins` array in the
     `all_built_in_tools_categorize_to_some` test.
   - `mcp/src/runner.rs`: add `tools::delete_file(scope.clone())`,
     `tools::move_file(scope.clone())`, `tools::patch_lines(scope.clone())` to the
     `build_registry` tool vec (`runner.rs:142-150`).

3. **Remove the two prod `unwrap`s** in `search.rs` `format_output`. Replace the
   `is_empty()`/`unwrap()` pairs with `map_or`:
   ```rust
   let before_start = hit.context_before.first().map_or(hit.line, |t| t.0);
   // ...
   last_after_end = hit.context_after.last().map_or(hit.line, |t| t.0);
   ```

4. **Fix formatting.** Per CLAUDE.md, do **not** run `cargo fmt --all` (writing
   form) ‚Äî run `rustfmt` only on the phase-touched files:
   `rustfmt executor/src/tools/{find_files,search,write_file,patch_lines,delete_file,move_file}.rs`.

5. Re-run all four gates as separate invocations and confirm green, including the
   new tools' tests now actually executing (the run count must jump well above 766
   once `delete_file`/`patch_lines`/`move_file` tests compile in).

# Bug 1 on phase-01: missing CLI-parse test for the `run-phase` subcommand

**Severity:** minor
**Status:** open
**Filed:** 2026-05-29

## What's wrong

Phase-01 Acceptance criterion 5 explicitly required a **CLI-parse test** for the
new `rexymcp run-phase` subcommand:

> `rexymcp run-phase …` parses its args, builds `Config`, and calls `run_phase`
> (exercised by a CLI-parse test; the full network path is `#[ignore]`-gated or
> simply not unit-tested — it needs a live endpoint).

The 10 added tests in `mcp/src/runner.rs` cover `parse_phase_doc` (5),
`derive_phase_id` (2), `build_registry` (1), and `run_phase_with` (2). **None
asserts that the `run-phase` clap subcommand parses its expected flags.**

The runtime path works (clap-derive's compile-time checks catch shape errors),
but the explicit checkbox is unticked, and the Update Log's "all ticked above"
is therefore inaccurate. The criterion's intent was to give the new
`mcp/src/main.rs` wiring (the `RunPhase { config, phase_doc, repo, model }`
subcommand) an automated reachability assertion so it doesn't silently break
when phase-02 reworks `main.rs` to add the MCP server.

## How to fix

Add one `#[test]` (anywhere reachable from `mcp` — e.g. in `mcp/src/runner.rs`
under `#[cfg(test)] mod tests`, or a new tiny `#[cfg(test)] mod cli` block in
`mcp/src/main.rs`) that calls `<Cli as clap::Parser>::try_parse_from(...)` with
the expected arg vector and asserts the resulting `Commands::RunPhase` variant
has the expected `config` / `phase_doc` / `repo` / `model` fields.

```rust
use clap::Parser;
let cli = Cli::try_parse_from([
    "rexymcp", "run-phase",
    "--config", "rexymcp.toml",
    "--phase-doc", "docs/dev/milestones/M5-mcp-server/phase-01-phase-runner.md",
    "--repo", "/tmp/repo",
    "--model", "qwen2.5-coder",
]).unwrap();
match cli.command {
    Some(Commands::RunPhase { config, phase_doc, repo, model }) => {
        assert_eq!(config, PathBuf::from("rexymcp.toml"));
        assert_eq!(phase_doc, PathBuf::from("docs/dev/milestones/M5-mcp-server/phase-01-phase-runner.md"));
        assert_eq!(repo, PathBuf::from("/tmp/repo"));
        assert_eq!(model.as_deref(), Some("qwen2.5-coder"));
    }
    other => panic!("expected RunPhase, got {other:?}"),
}
```

Also add a second case asserting `--model` is optional (omitted → `None`), and
that an invalid arg shape returns `Err` from `try_parse_from`.

If the test lives in `runner.rs`, expose `Cli` / `Commands` from `main.rs` via a
small `pub` (or `pub(crate)`) seam so the test can reach them — or put the test
beside them in `main.rs`. Either is fine; pick whichever needs less surface
change.

## Why this matters

This is bug-class because the criterion was explicit. The fix is small (one or
two tests), but the discipline matters: an unticked acceptance criterion is the
phase-doc's contract not being met, regardless of how trivial the gap is.

Phase-01 status flips back to `in-progress` until this is resolved + re-reviewed.

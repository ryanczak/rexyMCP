# Bug 1 on phase-02: request-shaping logic is untested (spec test missing)

**Severity:** minor
**Status:** verified
**Filed:** 2026-05-28
**Verified:** 2026-05-28 — `ab9724e`. `build_chat_body` + `convert_messages`
extracted as pure fns; 6 new tests assert `stream: true` + model, `tool_choice`
none/auto toggle (incl. `Some(&[])`→none), and `convert_messages` for both the
tool-results and tool-calls shapes; 38 tests pass, gates clean; the completion
Update Log's E2E line corrected (overstatement acknowledged).

## What's wrong

The phase-02 test plan required:

> An OpenAI request-shaping test: build a client, capture the JSON body it would
> send … assert `stream: true` and the model field.

No such test exists. In `executor/src/ai/backends/openai.rs`:

- The request-body assembly inside `chat()` (`openai.rs:112-145`) — the
  `stream: true` flag, the `model` field, and the `tool_choice` `none`/`auto`
  toggle — is **untested**.
- `OpenAiClient::convert_messages` (`openai.rs:48-84`) — which maps a
  `tool_results` message to `role: "tool"` and a `tool_calls` message to
  `role: "assistant"` with a `tool_calls` array — is **untested**.

The six `openai.rs` tests cover `parse_openai_usage`, `emit_tool_call_generic`,
and `render_openai_tools` — all good, but none touch the request body or message
conversion.

Separately, the completion Update Log's End-to-end section states the
"request-body builder (exercised by a unit test)" is a verified real artifact.
There is no request-body test, so that line **overstates coverage** and should be
corrected.

## What should happen

The request-shaping logic runs on every executor turn (M4 onward), so it needs
direct hermetic coverage, and the spec explicitly asked for it. The body
assembly and message conversion should be unit-testable without a live socket.

## How to fix

In `executor/src/ai/backends/openai.rs`:

1. Extract the body assembly from `chat()` into a pure function, e.g.:
   ```rust
   fn build_chat_body(
       model: &str,
       system: &str,
       messages: Vec<Message>,
       tools: Option<&[ToolSchema]>,
   ) -> serde_json::Value
   ```
   Move the `convert_messages` + system-merge + body-json logic (currently
   `chat()` lines ~112-145) into it; have `chat()` call it, then POST the result.
   `convert_messages` can stay a method or become a free fn — your call.
2. Add hermetic unit tests (pin behavior, not names/count):
   - body has `stream == true` and `model` equal to the given model.
   - `tool_choice == "none"` when `tools` is `None` or empty; `"auto"` when
     non-empty (and a `tools` array is present).
   - `convert_messages` (or the builder) maps a message carrying `tool_results`
     to a `role: "tool"` entry with the `tool_call_id`, and a message carrying
     `tool_calls` to a `role: "assistant"` entry with a `tool_calls` array.
3. Correct the phase-02 completion Update Log's End-to-end line so it accurately
   names what is covered (remove/fix the "request-body builder exercised by a
   unit test" claim, or make it true by adding the test above).

No new dependencies. Scope limited to `openai.rs` (and the phase doc's Update
Log). Do not touch the streaming loop's behavior.

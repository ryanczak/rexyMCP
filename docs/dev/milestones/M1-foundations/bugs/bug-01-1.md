# Bug 1 on phase-01: env-override test uses `unsafe` and mutates real process env

**Severity:** major
**Status:** open
**Filed:** 2026-05-28

## What's wrong

`executor/src/config.rs:199-206` — the `env_override_beats_toml` test mutates the
real process environment through `unsafe`:

```rust
// SAFETY: test-only env mutation, no concurrent threads
unsafe {
    std::env::set_var("REXYMCP_MODEL", "model-b");
}
let cfg = Config::load_with_env(&path).unwrap();
// SAFETY: test-only env cleanup
unsafe {
    std::env::remove_var("REXYMCP_MODEL");
}
```

This violates two standards:

- **STANDARDS.md §1 (Definition of Done):** "No new `unsafe` blocks. (If you
  think you need one, stop and report a blocker.)" The phase doc did not
  authorize `unsafe`. In edition 2024 `std::env::set_var`/`remove_var` are
  `unsafe`, so the test backed into an unauthorized `unsafe` block instead of the
  executor filing a blocker.
- **STANDARDS.md §3.3:** tests must be hermetic and deterministic — "no real
  wall-clock, no shared global state." The process environment is shared global
  state; mutating it in a parallel test runner is exactly the hazard the standard
  forbids.

**Root cause is partly the spec.** The phase-01 test plan said "set/remove env
vars within the test," which in edition 2024 forces `unsafe`. The fix below
removes the conflict by making env application injectable, so the test never
touches real process env. (Recorded as calibration data — see the M1 README
Notes; first occurrence, not yet folded into STANDARDS/WORKFLOW.)

## What should happen

`Config`'s env application should be testable with a fake getter, so the override
behavior is covered hermetically with no `unsafe` and no real env mutation. The
public `apply_env` keeps reading the real environment for production use.

## How to fix

In `executor/src/config.rs`:

1. Extract an injectable helper:
   ```rust
   /// Apply REXYMCP_* overrides using the supplied getter.
   /// `apply_env` calls this with the real process environment.
   pub fn apply_overrides(&mut self, get: impl Fn(&str) -> Option<String>) {
       if let Some(v) = get("REXYMCP_PROVIDER") { self.executor.provider = v; }
       if let Some(v) = get("REXYMCP_MODEL")    { self.executor.model = v; }
       if let Some(v) = get("REXYMCP_BASE_URL") { self.executor.base_url = v; }
       if let Some(v) = get("REXYMCP_API_KEY")  { self.executor.api_key = Some(v); }
   }
   ```
2. Reduce `apply_env` to:
   ```rust
   pub fn apply_env(&mut self) -> Result<()> {
       self.apply_overrides(|k| std::env::var(k).ok());
       Ok(())
   }
   ```
   (Keep the `Result` return — a future numeric field may need to fail with
   `Error::Config`, per the original spec §4.)
3. Rewrite the override test to drive `apply_overrides` with an in-memory map
   (e.g. a `HashMap<&str,String>` closure) instead of mutating the environment.
   Pin the **behavior** (an override value beats the TOML-loaded value), not the
   test name or placement. Remove both `unsafe` blocks.

No new dependencies. No change to the public `Config` shape beyond adding
`apply_overrides`.

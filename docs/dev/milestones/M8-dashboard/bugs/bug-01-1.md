# Bug 1 on phase-01: dashboard event loop uses a different crossterm than ratatui's backend

**Severity:** major
**Status:** open
**Filed:** 2026-06-02

## What's wrong

`mcp/Cargo.toml` declares `crossterm = "0.28"` while `ratatui = "0.30"`. The
semver requirement `"0.28"` resolves to `>=0.28.0, <0.29.0`, so it does **not**
unify with the crossterm `ratatui 0.30` uses — ratatui pulls `crossterm 0.29.0`
(via `ratatui-crossterm v0.1.0`). The tree therefore contains **two** crossterm
versions:

```
$ grep -A2 'name = "crossterm"' Cargo.lock
name = "crossterm"
version = "0.28.1"
--
name = "crossterm"
version = "0.29.0"
```

`mcp/src/dashboard.rs:115` (`use crossterm::event::{self, Event, KeyCode,
KeyEventKind};`) and the `event::poll` / `event::read` calls at
`mcp/src/dashboard.rs:122-124` bind to the **0.28.1** direct dependency, whereas
`ratatui::init()`, `ratatui::restore()`, and `terminal.draw` at
`mcp/src/dashboard.rs:104-133` drive raw mode, the alternate screen, and the
SIGWINCH/resize machinery through **0.29.0**. The input subsystem and the
terminal backend are two independent crates with independent global state. This
is a correctness risk for the phase's core feature — the event loop and the
"exits cleanly and restores the terminal" acceptance criteria — and it ships a
duplicate dependency.

The completion Update Log note ("the explicit `crossterm = "0.28"` … resolved to
`0.29` via ratatui's dependency resolution — this is fine") is **incorrect**:
`"0.28"` cannot resolve to `0.29`; cargo kept both.

## What should happen

A single, version-matched crossterm: the crate the event loop polls must be the
same crate (and version) ratatui drives the terminal with. ratatui 0.30 (the
authorized dependency) requires crossterm 0.29, so the direct dependency must be
`crossterm = "0.29"`. The phase doc's original `crossterm = "0.28"` pin (Task 1
and the first acceptance criterion) was an architect error — it has been
corrected to `0.29` in the phase doc as part of this bounce, so there is no
longer a contradiction between the authorized ratatui 0.30 and the crossterm pin.

## How to fix

1. In `mcp/Cargo.toml`, change `crossterm = "0.28"` to `crossterm = "0.29"`.
2. Run `cargo build`; confirm `Cargo.lock` now holds a **single** crossterm
   entry (`0.29.x`) and no `0.28.x`.
3. No source change is required in `dashboard.rs` — `crossterm::event::{Event,
   KeyCode, KeyEventKind}` are stable across 0.28→0.29 for the code as written.
   Re-run the full command set.
4. Update the completion Update Log note to reflect the corrected, single-version
   resolution (remove the "resolved to 0.29 … this is fine" claim).

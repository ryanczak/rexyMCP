# Bug 1 on phase-05: non-hermetic test makes a real network call

**Severity:** major
**Status:** open
**Filed:** 2026-06-02

## What's wrong

The new test `is_retriable_transport_true_for_reqwest_error`
(`executor/src/ai/backends/openai.rs`) constructs its `reqwest::Error` by making
a **real outbound network request**:

```rust
#[tokio::test]
async fn is_retriable_transport_true_for_reqwest_error() {
    // Construct a real reqwest::Error by attempting a connection to an
    // unroutable address; wrap it in anyhow and assert it downcasts.
    let reqwest_err = reqwest::get("http://10.255.255.1:1").await.unwrap_err();
    let wrapped: anyhow::Error = reqwest_err.into();
    assert!(is_retriable_transport(&wrapped), "...");
}
```

This violates `STANDARDS.md` §3.3 ("Tests are **hermetic**: no real network …")
and §4 ("no real network"). Observed cost: the executor test suite now runs in
**~30.6 s wall-clock with only ~0.67 s CPU** — ~30 seconds blocked waiting for the
connection to `10.255.255.1:1` to time out. (Before this phase the suite was ~6 s.)
It is also non-deterministic — behaviour depends on the host's network/routing and
the connect-timeout, and it would hang or flake in a sandboxed CI environment with
no network.

The production code under test is correct; only the test is at fault.

## What should happen

The test must obtain a `reqwest::Error` **without any network I/O**, so the suite
stays hermetic and fast. A `reqwest::Error` is readily constructed synchronously
by building a request with a URL that fails to parse — no socket is opened.

## How to fix

Replace the network call with a synchronous, no-I/O construction. A relative URL
(no scheme/base) fails at request-build time with a `reqwest::Error`:

```rust
#[test] // no longer needs to be async / #[tokio::test]
fn is_retriable_transport_true_for_reqwest_error() {
    // A URL that fails to parse yields a reqwest::Error with no network I/O.
    let reqwest_err = reqwest::Client::new()
        .get("not-a-url")          // relative URL without a base → build error
        .build()
        .unwrap_err();
    let wrapped: anyhow::Error = reqwest_err.into();
    assert!(
        is_retriable_transport(&wrapped),
        "a reqwest::Error should downcast and be retriable"
    );
}
```

Notes for the executor:
- `is_retriable_transport` only checks `downcast_ref::<reqwest::Error>()`, so **any**
  `reqwest::Error` validates the classifier — a build/parse error works exactly as
  well as a transport error for this test's purpose.
- If `"not-a-url"` does not produce a synchronous build error in this reqwest
  version, any URL that fails to parse works (e.g. `"http://"` empty-host, or a URL
  with illegal characters). Confirm the chosen form returns `Err` from `.build()`
  **without** awaiting a send.
- Drop `#[tokio::test]` → `#[test]` (no async needed once the network call is gone).
- Also: on the re-dispatch, complete the closeout the run skipped — flip the phase
  doc `Status:` to `review`, fill the completion Update Log entry, and commit.
